# 모바일 정보 밀도 설계

## 목표

Aura-board 모바일 앱은 웹 화면의 카드 크기와 모달 모양을 축소 복제하지 않는다.

같은 데이터와 권한을 사용하되, 웹에서 넓은 화면을 통해 얻던 **판단 능력**을 모바일에서도 유지하는 것이 목표다.

- 현재 무엇이 진행 중인지 빠르게 파악한다.
- 여러 보드·주제·카드의 상태를 한 화면에서 비교한다.
- 필요한 항목을 선택한 뒤 상세 콘텐츠와 입력에 집중한다.
- 상세 기능과 저장 결과는 웹과 동일하게 유지한다.

이를 이 문서에서는 **판단 동등성**이라고 부른다.

## 문제였던 구조

### 보드 목록

정사각형 썸네일 카드를 2열로 보여 주어 제목과 유형은 확인할 수 있었지만, 다음 정보는 비교하기 어려웠다.

- 지금 참여해야 하는 게임이나 모둠 활동
- 수업 보드와 놀이 보드의 구분
- 보드별 카드 수
- 검색과 우선순위

### 일반 카드 보드

웹 카드의 이미지·본문·링크·작성자·반응을 한 카드 안에 그대로 축소했다. 카드 하나는 충실했지만 한 화면에 보이는 카드 수가 줄어 전체 흐름을 파악하기 어려웠다.

### 주제별 보드

데스크톱 칼럼을 가로 스크롤로 그대로 옮겼다. 모바일에서는 한두 칼럼만 보여서 주제별 카드 수와 활동량을 한 번에 비교할 수 없었다. 칼럼 안에 다시 세로 스크롤이 있어 위치 감각도 약해졌다.

## 새 정보 구조

### 1. 보드 허브

기본 목록은 큰 썸네일 그리드가 아니라 작은 썸네일과 상태를 포함한 압축 행이다.

첫 화면에서 다음을 함께 보여 준다.

- 전체 보드 수
- 지금 참여하거나 선택해야 하는 보드 수
- 수업·놀이 보드 수
- 전체 카드 수
- 보드별 현재 상태와 카드 수

필터는 `지금 할 것 / 수업 / 놀이 / 전체`로 구성한다. 진행 중인 게임이나 모둠 선택이 필요한 보드가 있으면 `지금 할 것`을 기본으로 연다.

### 2. 일반 카드 보드

좁은 화면에서는 압축 목록을 기본으로 사용한다.

- 카드 제목과 핵심 본문
- 작은 미디어 미리보기
- 작성자와 날짜
- 내 카드 여부
- 좋아요·댓글 수
- 소속 섹션

필요하면 `카드` 보기로 전환해 기존 전체 카드 표현을 사용할 수 있다. 검색과 `전체 / 내 카드 / 미디어 / 댓글 있음` 필터를 제공한다.

### 3. 주제별 보드

가로 칼럼 대신 `주제 개요 → 선택한 주제의 카드 목록` 구조로 바꾼다.

주제 개요에서는 모든 주제를 타일로 비교한다.

- 카드 수
- 내 카드 수
- 좋아요·댓글 수
- 가장 최근 카드

주제를 선택하면 해당 주제의 압축 카드 목록으로 들어간다. 카드 작성과 상세 모달은 기존 API·권한 계약을 그대로 사용한다.

## 데이터 원칙

- 새 API와 DB 필드를 추가하지 않는다.
- 기존 `/api/student/me`와 `/api/student/board/:slug` 응답을 파생 모델로 정리한다.
- Broadcast 기반 보드 갱신과 카드 작성·수정·삭제 계약을 유지한다.
- 서버가 내려준 섹션 시각 순서를 모바일이 다시 해석하지 않는다.
- 요약 값은 탐색을 돕는 정보이며 권한이나 저장 상태의 원본으로 사용하지 않는다.

## 검증 기준

- 학생이 보드 목록에서 현재 진행 중인 활동을 스크롤 없이 찾을 수 있다.
- 모바일 한 화면에서 여러 보드의 상태와 카드 수를 비교할 수 있다.
- 일반 카드 보드에서 전체 흐름을 목록으로 보고 원할 때 원본 카드 표현으로 전환할 수 있다.
- 주제별 보드에서 가로 스크롤 없이 모든 주제의 활동량을 비교할 수 있다.
- 기존 카드 작성, 상세, 좋아요·댓글, 수정·삭제, Realtime 갱신이 유지된다.
- `npm run typecheck`, `npm run design:check`, 관련 Vitest를 통과한다.
