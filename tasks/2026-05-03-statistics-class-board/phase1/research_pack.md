# Phase 1 — Research Pack

## 개요

통계활용대회 학급보드 기능의 UX/정책 벤치마크 분석. 5개 제품(Padlet, Miro, Google Classroom, Linear, Notion)을 대상으로 브레이크아웃룸, 단계별 미션, 교사 승인 게이트, 상태 트래킹, 질문 사다리 입력, LLM 피드백, 포스터 의뢰서 총 7개 패턴을 추출하고 장단점을 분석했다.

---

## 1. 기존 제품 (Base Product): Padlet

### 관찰 내용
- **카드 기반 자유 배치**: 사용자가 텍스트, 이미지, 파일, 링크를 카드 형태로 보드에 배치
- **실시간 협업**: 다중 사용자가 동시에 카드 추가/이동/편집 가능 (WebSocket 기반)
- **레이아웃 모드**: 그리드, 캔버스(프리폼), 스트림, 지도 등 다양한 레이아웃
- **권한 모델**: 소유자(Owner) / 중재자(Moderator) / 참여자(Contributor) / 읽기전용(Viewer)
- **反應**: 카드에 좋아요, 평점, 투표 등 반응 추가 가능

### 학급보드 적용 분석
- **장점**: 기존 코드베이스 확장이므로 인프라 재활용 가능. 실시간 협업 엔진 그대로 사용.
- **단점**: Padlet의 권한 모델은 '교사-학생' 역할보다는 '소유자-참여자'로 설계됨. 교사 승인 게이트와 같은 교육 특화 워크플로우가 없음.
- **결정**: 기존 보드/카드/권한 모델을 확장. 교사(Teacher) / 학생(Student) 역할을 추가하고, 미션 단위 승인 게이트를 새로 구현.

## 2. 브레이크아웃룸 레퍼런스: Miro

### 관찰 내용
- **브레이크아웃룸**: 메인 보드에서 팀별 하위 보드(프레임)로 전환. 각 팀은 독립적 캔버스를 가짐
- **메인 보드 오버뷰**: 모든 팀의 프레임을 한 화면에 축소 표시. 교사(진행자)가 전체 진행 상황 확인
- **타이머**: 팀별 작업 시간 제한 및 알림
- **투표**: 팀별 또는 전체 대상 투표 진행

### 학급보드 적용 분석
- **장점**: Miro의 프레임 전환 UX는 '팀별 집중'과 '전체 오버뷰'를 동시에 만족. 메인 보드에서 팀 카드를 클릭하면 팀 보드로 확장되는 패턴이 직관적.
- **단점**: Miro는 완전한 프리폼 캔버스라 구조화된 미션 입력이 어려움. 우리는 구조화된 폼(질문 사다리, 설문)이 필요함.
- **결정**: Miro의 '프레임 전환' 개념은 참고하되, 팀 보드 내부는 구조화된 미션 패널로 구성. 메인 보드에서는 팀별 상태 카드만 표시.

## 3. 교사-학생 워크플로우 레퍼런스: Google Classroom

### 관찰 내용
- **역할 분리**: 교사는 과제 생성/채점/반환, 학생은 제출/확인
- **과제 라이프사이클**: 할당됨 → 제출됨 → 채점 중 → 반환됨
- **진행 상황**: 교사용 대시보드에서 학생별 제출/미제출 상태 확인
- **승인/채점**: 교사가 '반환'하기 전까지 학생은 수정 불가(잠금). 교사 피드백 후 다시 수정 가능
- **기한**: Due Date 지나면 자동 잠금(교사 설정)

### 학급보드 적용 분석
- **장점**: 교사 승인 게이트(미션 2,3,6,8)의 정책적/권한적 레퍼런스가 명확. '제출 → 승인 → 다음 단계' 플로우가 이미 검증된 UX.
- **단점**: Google Classroom은 1-way 제출(학생→교사) 중심. 우리는 '질문 사다리'처럼 학생이 반복 수정하고, 교사가 중간중간 피드백하는 iterative 플로우가 필요함.
- **결정**: 승인 게이트 구조는 Google Classroom 방식을 따르되, '승인 요청 → 피드백 → 수정 → 재승인'의 iterative 사이클을 추가.

## 4. 상태 트래킹 레퍼런스: Linear

### 관찰 내용
- **상태 파이프라인**: Backlog → Todo → In Progress → In Review → Done
- **승인 게이트**: 'In Review' 상태에서만 'Done'으로 전환 가능. Reviewer 승인이 필요.
- **자동화**: 상태 변경 시 트리거(알림, 할당자 변경 등) 설정 가능
- **사이클/마일스톤**: 이슈를 사이클(주/2주 단위)로 그룹화하여 진도 관리
- **키보드 단축키**: `cmd+k` 상태 변경, 빠른 필터링

### 학급보드 적용 분석
- **장점**: 상태 기반 워크플로우가 시각적으로 명확. '승인 완료' 상태가 다음 미션 잠금 해제의 트리거가 되는 구조가 직관적.
- **단점**: Linear는 소프트웨어 팀을 위한 도구라 초등학생 대상 문구/복잡도가 맞지 않음. 키보드 단축키 등은 과함.
- **결정**: 5단계 상태(⬜시작전 🟡수정중 🟢승인완료 🔵교사제작중 ✅완료)를 미션별로 관리. 상태 변경은 버튼 기반(드롭다운). 자동화는 교사 승인 시 다음 미션 잠금 해제로 단순화.

---

## 5. 구조화된 폼/템플릿 레퍼런스: Notion

### 관찰 내용
- **데이터베이스 보드**: 속성(Property) 기반으로 테이블/보드/갤러리/캘린더 뷰 전환
- **템플릿**: 페이지 템플릿을 미리 만들고, 새 페이지 생성 시 템플릿 적용
- **인라인 AI**: 페이지 내에서 `/ai` 명령으로 AI에게 글쓰기/요약/번역 요청
- **계층적 페이지**: 페이지 안에 페이지를 무한히 중첩 가능

### 학급보드 적용 분석
- **장점**: Notion의 템플릿 개념을 '미션 템플릿'에 적용 가능. 매번 새 프로젝트(통계활용대회) 시작 시 11개 미션 템플릿을 자동 생성. 인라인 AI는 질문 사다리 피드백 UX의 레퍼런스.
- **단점**: Notion의 자유도가 너무 높아 초등학생이 사용하기엔 복잡함. 속성/필터 개념은 숨김.
- **결정**: 미션은 템플릿 기반으로 자동 생성. 질문 사다리 입력은 Notion의 인라인 AI처럼 입력 영역 하단에 피드백 풍선 표시. 계층 구조는 단순화(보드 → 팀 → 미션).

## 6. 핵심 UX 패턴 종합 결정

| 패턴 | 결정 | 근거 |
|---|---|---|
| 브레이크아웃룸 | **ADOPT** | 기존 코드베이스에 이미 존재. 확장 사용. |
| 단계별 미션 가이드 | **ADAPT** | 수직 스텝퍼 + 미션 카드 혼합. 11단계를 한 화면에 표시하되 현재 미션만 확장. |
| 교사 승인 게이트 | **ADOPT** | 핵심 요구사항. 알림/대기 상태 시각화로 보완. |
| 상태 기반 보드 트래킹 | **ADOPT** | 교사용 대시보드(표) + 학생용 보드(팀 카드) 이중 UI. |
| 질문 사다리 입력 폼 | **ADAPT** | 아코디언 형태. 한 단계씩 확장. 예시는 풍선 도움말. |
| LLM 피드백 인라인 | **ADOPT** | 기존 API 활용. 교사 승인과 병행하여 LLM은 '조언' 수준. |
| 포스터 의뢰서 제출 폼 | **ADOPT** | 미션 1~8 산출물 자동 연동으로 중복 입력 최소화. |

## 7. 리스크 및 완화책

| 리스크 | 영향 | 완화책 |
|---|---|---|
| 11단계 미션이 너무 많아 학생 이탈 | 높음 | 현재 미션 3개만 노출(이전/현재/다음). 나머지는 '더 보기'로 접기. |
| 교사 승인 지연으로 진행 정체 | 높음 | 승인 요청 시 푸시 알림 + 대기 중에도 이전 미션 수정 가능(재제출). |
| LLM 피드백 오류 | 중간 | '교사 확인 필요' 배지 표시. 학생에게 'AI 조언일 뿐' 문구 명시. |
| 팀 수 30팀+ 시 대시보드 성능 | 중간 | 가상 스크롤 적용. 상태별 필터링 기본 제공. |
| 모바일에서 브레이크아웃룸 전환 불편 | 중간 | 팀 보드 진입 시 전체 화면 모드. 하단 탭으로 미션 전환. |
